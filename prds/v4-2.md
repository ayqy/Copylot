# 代码块复制稳定性专项

## 目标
实现代码块场景中“复制后无需返工”的最小可交付增量，优先修掉缩进、行号和复制按钮噪声残留等会直接阻碍第二次复用的问题，并沉淀可导出、可复核的商业化证据，使其达到可发布验收标准

## ULTRA IMPORTANT 第一原则：任务开始后，不要停下来，直到完成目标
**所有细节，都由你来定，判断依据是根据核心目标决定应该如何做，而不要停下来问我**

## 具体任务
1、更新 `src/shared/code-block-cleaner.ts`、`src/shared/content-processor.ts`、`src/content/content.ts` 与相关调用链，只处理代码块场景，补齐对缩进保留、可识别行号剥离、复制按钮噪声整行清理的稳定性修复，确保用户从文档站、教程页或 README 类页面复制代码后无需手工返工即可直接粘贴到编辑器或 AI。
2、更新 `e2e/code-block-flow.spec.ts`、`e2e/fixtures/code.html` 与必要的单元/集成测试入口，覆盖“悬停复制代码块”“整页转换保留 fenced code”“缩进与空行保持”“行号与复制按钮噪声不会残留”四类核心回归，并创建 `docs/test-cases/v4-2.md` 作为手工与自动化统一验收用例文档。
3、创建并维护 `docs/evidence/v4-2/code-block-reuse-audit.md`、`docs/evidence/v4-2/code-block-commercialization-proof.md`、`docs/evidence/v4-2/code-block-sample.txt` 与 `docs/reports/v4-2-report.md`，明确记录本轮修复前后对“复制后无需返工”的影响、可验证的复用价值证据、测试结果、涉及文件范围与可口头汇报摘要，且整体任务规模控制在 1-3 天内可完成。

## 验收标准
* 修复上述所有问题，并测试通过，达到可发布状态
* 实现以上所有需求，并测试通过，达到可发布状态
* `docs/test-cases/v4-2.md` 中定义的自动化与手工用例均可执行，且 `bash scripts/test.sh` 退出码为 `0`
* 至少 1 条代码块悬停复制路径与 1 条整页转换路径能够稳定产出可直接粘贴的代码结果，复制结果保留原始缩进与内部空行，不残留可识别行号或复制按钮噪声
* `docs/evidence/v4-2/code-block-reuse-audit.md` 必须明确说明字段来源、样例页面、问题口径、回归步骤与“无需返工”的判定标准，`docs/evidence/v4-2/code-block-sample.txt` 必须可用于人工对比修复前后复制结果
* `docs/evidence/v4-2/code-block-commercialization-proof.md` 必须提供可验证的商业化证据，例如“代码块场景复制结果可直接用于 AI / 编辑器的样例”“代码块专项回归通过后可继续承接第二次 `copy_success` 的证明路径”“对减少返工、提高复用意愿的量化判断口径”，避免只证明代码修好了却无法证明其推动了真实复用
* `docs/reports/v4-2-report.md` 必须逐项说明具体任务完成状态、效果、涉及目录文件与测试结果，能够直接用于口头汇报
