# 问题修复和功能优化（V1-33 CWS 发布流程防呆：publish:cws 强制生产回归 + 支持 dry-run）

## 目标
在不新增任何扩展权限、不改变现有复制/Prompt/引导/口碑/商业化功能逻辑的前提下，补齐 `prds/v1.md` 中“测试用例与发布准备”的最后一段发布工程化闭环：让 `npm run publish:cws` 成为**安全、可重复、可审计**的 Chrome Web Store 发布入口（强制生产回归与产物校验；提供无需凭据的 dry-run 演练），使之达到验收标准。

## 具体任务
1、`publish:cws` 强制走“生产回归 + 产物校验”，杜绝误发开发构建
  - 修改：`scripts/chrome-webstore.ts`
  - 新增参数：`--dry-run`
    - `--dry-run`：不做任何上传/发布网络调用；允许在无 `.env` 凭据的环境中演练完整发布前置流程（仍需打印缺失的凭据项，便于真实发布前补齐）。
    - 非 `--dry-run`：保持现有上传并发布行为，但必须在任何网络调用前完成全部 preflight 与回归门禁。
  - 发布前置门禁（任何失败都必须退出非 0，并打印清晰原因）：
    - 强制执行 `bash scripts/test.sh`（包含 lint/type-check/check-i18n/unit-tests/build:prod/verify-prod-build），作为“允许发布”的唯一门禁。
    - 读取根目录 `manifest.json` 的 `version`，并校验 `dist/manifest.json` 的 `version === manifest.json.version`（避免错包/旧产物）。
    - 打包策略：基于当前 `dist/` 重新生成 `plugin-${version}.zip`（若已存在同名 zip，必须先删除再生成），确保 zip 与本次回归后的产物一致。
  - 行为约束：
    - 严禁调用 `npm run build`（开发构建）；只能依赖 `bash scripts/test.sh` 产出的生产 `dist/`。
    - 除生成/覆盖 `plugin-${version}.zip` 外，不得创建/修改任何 git commit/tag，不得改写 `manifest.json` 版本号。

2、补齐发布文档口径：明确 “本地发布” vs “CWS 发布” 的推荐顺序与 dry-run 演练方式
  - 修改：`BUILD-PRODUCTION.md`
  - 要求：
    - 新增章节（或小节）说明推荐顺序：
      - 本地发布（版本 bump/回归/产物）：`npm run publish`
      - CWS 发布（上传/发布）：`npm run publish:cws`
    - 明确给出无需凭据的演练命令：`npm run publish:cws -- --dry-run`，并解释其不会发生任何上传/发布。

3、补齐测试/用例闭环（让验收可执行、可回归）
  - 新增测试用例文档：`docs/test-cases/v1-33.md`
  - 文档必须覆盖（至少）：
    - 用例 A（dry-run 可演练且无网络副作用）：
      - 在未配置 `.env` 的环境中执行 `npm run publish:cws -- --dry-run`
      - 期望：不会尝试上传/发布；会执行 `bash scripts/test.sh`；生成/覆盖 `plugin-${version}.zip`；并打印缺失的 CWS 凭据项提示
    - 用例 B（非 dry-run 缺少凭据时失败可审计）：
      - 执行 `npm run publish:cws`（不提供凭据）
      - 期望：在任何网络调用前失败退出非 0，并提示缺少的环境变量；不得出现“上传成功/发布成功”等误导信息
    - 用例 C（产物一致性）：
      - 校验 `dist/manifest.json` 的 `version` 与 `manifest.json` 一致；zip 文件名中的版本号与之完全一致
    - 自动化回归记录：
      - 记录一次 `bash scripts/test.sh` 的执行结论与日期（作为发布门禁的审计证据）
