# V1-47 发布阻塞解除 v2：`publish:cws` 支持 `socks5://` 代理 + 网络可达性预检取证 测试用例

## 目标
验证子 PRD `prds/v1-47.md` 的验收点：在不新增权限、不泄露发布凭据、不采集用户复制内容的前提下，将 `npm run publish:cws` 的网络代理能力从“仅 http/https”升级为“支持 socks5/socks5h”，并新增可审计的网络可达性预检（Preflight）与证据落盘，使失败原因可复制、可定位、可审计。

本轮新增/固化的交付物（验收对象）：
- 预检与诊断：`scripts/chrome-webstore.ts`（输出 Proxy Diagnostic Block + Preflight Report Block + Diagnostic Pack）
- 关键逻辑单测：`scripts/unit-tests.ts`
- 证据目录与索引：`docs/evidence/v1-47/` + `docs/evidence/v1-47/index.md`
- 简报：`docs/reports/v1-47-report.md`

## 用例 A（离线）：无代理环境的预检输出（FAIL 也必须可定位）

步骤：
1. 清理代理环境变量（示例）：
   - `unset CWS_PROXY HTTPS_PROXY HTTP_PROXY ALL_PROXY https_proxy http_proxy all_proxy`
2. 执行：`npm run publish:cws -- --dry-run`
3. 检查输出包含稳定标记：
   - `-----BEGIN CWS PROXY DIAGNOSTIC BLOCK-----` / `-----END CWS PROXY DIAGNOSTIC BLOCK-----`
   - `-----BEGIN CWS PREFLIGHT REPORT BLOCK-----` / `-----END CWS PREFLIGHT REPORT BLOCK-----`
   - `-----BEGIN CWS PUBLISH DIAGNOSTIC PACK-----` / `-----END CWS PUBLISH DIAGNOSTIC PACK-----`
4. 检查 Proxy Diagnostic Block 中：
   - `proxy.enabled=false`
5. 检查 Preflight 输出：
   - 每个依赖域名都有明确 `PASS/FAIL` 行
   - FAIL 时包含失败类型（如 `dns/timeout/tls/http_status/connection_refused/permission`）而不是只有 `fetch failed`
   - 输出中包含“Preflight 失败修复建议（最短可执行）”

期望：
- dry-run 退出码为 0（仍不进行 upload/publish）。
- 即使网络不可达，也能输出可复制的失败类型与修复建议。

## 用例 B（可选：有代理则测）：`http://` 代理的预检输出（PASS/FAIL 需可审计）

步骤：
1. 设置 HTTP 代理（示例）：
   - `export HTTPS_PROXY=http://127.0.0.1:7890`
   - `export NO_PROXY=localhost,127.0.0.1,::1`
2. 执行：`npm run publish:cws -- --dry-run`
3. 检查 Proxy Diagnostic Block 中：
   - `proxy.enabled=true`
   - `proxy.envKey` 为 `HTTPS_PROXY`（或更高优先级的 `CWS_PROXY`）
   - `proxy.urlMasked` 为 `http://127.0.0.1:7890`（不包含用户名密码）
4. 检查 Preflight：
   - 若代理可达且可访问 Google API：应为 `PASS`
   - 若代理不可用：应为 `FAIL` 且失败类型为 `connection_refused/timeout` 等可定位原因

期望：
- 预检结果可用于判断“代理是否真的生效 + 外网依赖是否可达”。

## 用例 C（重点）：`socks5://` 代理的预检输出（必须识别 socks5 并给出可定位失败类型）

步骤：
1. 设置 Socks5 代理（示例）：
   - `export HTTPS_PROXY=socks5://127.0.0.1:1080`
   - `export NO_PROXY=localhost,127.0.0.1,::1`
2. 执行：`npm run publish:cws -- --dry-run`
3. 检查 Proxy Diagnostic Block 中：
   - `proxy.enabled=true`
   - `proxy.urlMasked` 以 `socks5://` 开头（明确展示已识别并使用 socks5）
   - `fetch.dispatcher` 为 `undici.Agent (socks5 connector) (setGlobalDispatcher)`
4. 检查 Preflight：
   - PASS/FAIL 均可，但 FAIL 时必须输出可定位失败类型（DNS/timeout/connection_refused/TLS/HTTP status）

期望：
- 在仅提供 `socks5://127.0.0.1:PORT` 的环境下，不需要“转 HTTP 代理端口”的人工操作即可跑通同一套代理解析逻辑。

## 用例 D（重点）：`socks5h://` 远程 DNS（用于 DNS 失败场景）

步骤：
1. 将代理改为：
   - `export HTTPS_PROXY=socks5h://127.0.0.1:1080`
2. 执行：`npm run publish:cws -- --dry-run`
3. 若此前 `socks5://` 场景出现 `dns` 失败：
   - 本用例应能验证“远程 DNS”路径（至少输出中明确展示 `socks5h://` 已被识别并使用）

期望：
- 输出明确展示 `socks5h://` 已生效（用于降低本地 DNS 不可达/污染导致的阻塞概率）。

## 用例 E（安全）：诊断输出脱敏校验（禁止泄露 secret/token）

步骤：
1. 设置带用户名密码的代理（示例）：
   - `export CWS_PROXY=socks5://user:pass@127.0.0.1:1080`
2. 执行：`npm run publish:cws -- --dry-run`
3. 检查输出：
   - Proxy Diagnostic Block 中 `proxy.urlMasked` 不包含 `user/pass`
   - 输出中不得出现 `CWS_CLIENT_SECRET`/`CWS_REFRESH_TOKEN` 的真实值（只允许显示“已设置/未设置”）

期望：
- 所有诊断输出均可被贴到 Issue/简报，且无敏感信息泄露风险。

## 用例 F（需网络可达）：真实发布前的预检阻断（非 dry-run）

步骤：
1. 在代理可用/可访问 Google API 的环境下执行：`npm run publish:cws`
2. 若 Preflight FAIL：
   - 应在 upload/publish 前被阻断（退出码非 0）
   - 输出包含修复建议与完整诊断块（可复制）

期望：
- 网络不可达时不再“跑到 upload 阶段才失败且只有 fetch failed”，而是提前可审计地失败并指路。
