## 用例 A（失败不污染 Git）
目标：在一次可控的 `bash scripts/test.sh` 失败下，验证 `npm run publish` **不会污染 Git**，并会自动恢复 `manifest.json`。

预置条件：
- 创建临时分支（例如：`git checkout -b tmp/v1-32-case-a`）
- 工作区必须干净（`git status --porcelain` 为空）
- 制造一次可控的 `type-check` 失败（示例：引入一个可逆的 TS 类型错误并提交，保证工作区仍然干净）

步骤：
1. 运行：`npm run publish`
2. 在版本确认提示中输入 `y`（仅用于触发发布流程继续执行）

期望：
- `npm run publish` 在执行 `bash scripts/test.sh` 时失败并退出非 0
- 未产生任何新的 commit（仅应存在用来制造失败的那次临时提交）
- 未创建 `v${newVersion}` tag
- 未生成 `plugin-${newVersion}.zip`
- `manifest.json` 版本号已自动恢复（`git diff` 为空，且 `manifest.json` version 回到原值）

执行记录：
- 结果：通过
- 最近执行日期：2026-03-20
- 备注：回归失败后脚本会直接写回旧 manifest 原始内容；commit/tag/zip 均不会发生。

## 用例 B（成功路径产物校验）
目标：在干净工作区且 `bash scripts/test.sh` 可通过时，验证 `npm run publish` 能完成到 zip 生成，并满足产物一致性校验。

预置条件：
- 创建临时分支（例如：`git checkout -b tmp/v1-32-case-b`）
- 工作区必须干净（`git status --porcelain` 为空）
- `bash scripts/test.sh` 可通过

步骤：
1. 运行：`npm run publish`
2. 在版本确认提示中输入 `y`
3. 当脚本询问是否推送 tag / push commit 等联网步骤时，输入 `n` 结束流程（本用例只验证到 zip 生成即可）

期望：
- `bash scripts/test.sh` 全量通过（lint/type-check/check-i18n/unit-tests/build:prod/verify-prod-build）
- 生成 commit：`chore: bump version to ${newVersion}`
- 生成 tag：`v${newVersion}`
- 生成 zip：`plugin-${newVersion}.zip`
- `dist/manifest.json` 的 `version` 与 `newVersion` 一致（由 publish 脚本在打包前强校验）
- 针对该 `dist/` 执行 `bash scripts/verify-prod-build.sh` 自检通过

执行记录：
- 结果：通过
- 最近执行日期：2026-03-20
- 备注：本用例在 zip 生成后主动选择不 push，以避免联网步骤；本地 commit/tag/zip 均按约定生成。
