# 问题修复和功能优化（V1-17 Prompt 管理器排序能力：按最常用/最近使用排序 + 修正“最近使用”时间显示）

## 目标
在不新增权限、不新增任何联网能力、且不改变既有 Prompt 数据结构语义（`usageCount/lastUsedAt` 仅用于本地/同步的效率优化）的前提下，为 Options 的“私人 Prompt 管理器”补齐**可切换排序**能力（默认/最常用/最近使用），并修正卡片底部“最近使用”时间当前误用 `createdAt` 的问题：让用户能更快找到高频 Prompt、形成复用习惯，使之达到验收标准并可发布。

## 具体任务
1、Options 的 Prompt 工具条新增“排序方式”控件（稳定 DOM、可 i18n、默认不打扰）
  - 位置：`src/options/options.html` -> Prompts Tab -> `.toolbar` 的 `.filter-options` 内，紧邻现有 `#category-filter`
  - 新增排序下拉框（建议 `<select>`，避免引入复杂交互）：
    - 控件 ID：`#prompt-sort-select`
    - 可选项（value 固定，便于用例与未来扩展）：
      - `default`：默认（保持当前顺序/不额外排序，作为兼容兜底）
      - `most_used`：最常用（按 `usageCount` 降序）
      - `recent_used`：最近使用（按 `lastUsedAt` 降序；无 `lastUsedAt` 的视为“未使用过”并排在末尾）
  - i18n：
    - 新增/补齐 `_locales/en/messages.json` 与 `_locales/zh/messages.json` 文案 key（至少包含：下拉框 label/placeholder、三个选项的展示文案）
    - 确保 `npm run check-i18n` 通过
  - 约束：
    - 不新增任何权限
    - 不新增第三方依赖
    - 不新增任何自动弹窗/提示（仅提供 UI 控件）

2、落地排序逻辑与“最近使用”显示规则（只影响渲染顺序，不改写存储顺序）
  - 逻辑落地：`src/options/options.ts`
  - 状态管理：
    - 新增 `promptSortMode`（取值严格限定为 `default|most_used|recent_used`）
    - `#prompt-sort-select` 的 `change` 事件触发 `filterAndRenderPrompts()` 重新渲染
    - 选择记忆（最小可交付即可，二选一且必须在 PR 中固定一种实现）：
      - 方案 A：使用 `localStorage` 持久化（key 建议 `copylot_prompt_sort_mode`），仅影响本机
      - 方案 B：写入 `chrome.storage.sync`（需新增 `copilot_settings.promptSortMode` 字段并补齐默认值/迁移/单测）
  - 排序规则（必须可回归、可单测）：
    - 仅在 `filteredPrompts` 渲染前对其做**拷贝后排序**（避免影响 `allPrompts` 原始顺序与保存逻辑）
    - `default`：不排序（保持 filter 产出的原始顺序）
    - `most_used`：
      - 主排序：`(usageCount ?? 0)` 降序
      - 次排序：`title`（case-insensitive）升序（保证确定性）
    - `recent_used`：
      - 主排序：`lastUsedAt` 降序（无值视为 `0`，必须排在末尾）
      - 次排序：`(usageCount ?? 0)` 降序（提高“最近且常用”的稳定性）
      - 再次排序：`title`（case-insensitive）升序
  - 修正 Prompt 卡片底部时间显示（避免误导用户）：
    - 位置：`src/options/options.ts` -> `createPromptCard(...)`
    - 将当前的 `formatTimeAgo(prompt.createdAt)` 改为：
      - 若 `prompt.lastUsedAt` 存在：展示 `formatTimeAgo(prompt.lastUsedAt)`
      - 否则：展示一个明确的“未使用过”占位文案（需 i18n，例如 `promptNeverUsed`），不得继续用 `createdAt` 冒充“最近使用”
  - 约束：
    - 不新增/不修改任何埋点事件名（本需求仅为本地 UI 效率提升）
    - 不改变 Prompt 的写入逻辑与同步策略（仍由既有 `saveSettings({ userPrompts })` 路径负责）

3、补齐单测与测试用例文档（可执行、可回归、可发布）
  - 单测（最小但必须）：
    - 若为“排序逻辑”抽出纯函数（推荐，例如 `sortPrompts(prompts, mode)`），必须在 `scripts/unit-tests.ts` 增加断言覆盖：
      - `most_used`：按 `usageCount` 降序，缺省值视为 `0`，且有稳定次排序
      - `recent_used`：有 `lastUsedAt` 的排在前；无 `lastUsedAt` 的排在末尾；同样具备稳定次排序
      - `default`：保持输入顺序不变
    - 若实现“选择记忆”，需覆盖其兜底行为（非法值回退到 `default`）
  - 测试用例文档：
    - 新增并维护 `docs/test-cases/v1-17.md`，覆盖（至少）：
      - 用例 A（控件可见）：打开 Options -> Prompts Tab，`#prompt-sort-select` 可见且可切换三种模式
      - 用例 B（最常用排序）：构造 3 个 Prompt 且 `usageCount` 不同，切换到 `most_used` 后渲染顺序正确；并验证同 usageCount 时按标题稳定排序
      - 用例 C（最近使用排序）：触发任意 1 个 Prompt 使用（右键菜单或悬浮按钮），回到 Options 切换 `recent_used` 后该 Prompt 置顶；未使用过的 Prompt 必须排在末尾
      - 用例 D（时间显示正确）：已使用 Prompt 显示相对时间（基于 `lastUsedAt`）；未使用 Prompt 显示 `promptNeverUsed`（而非 createdAt）
      - 用例 E（选择记忆，如实现）：刷新 Options 页面后排序模式保持（localStorage 或 settings），并记录验证方式
      - 自动化回归：记录一次 `bash scripts/test.sh` 执行结果（命令 + 结论）
  - 验收前完成回归：确保不影响既有 Prompt 搜索/分类/批量操作/导入导出、Chat 服务管理、隐私面板、Pro/WOM/反馈入口等功能，达到可发布状态
