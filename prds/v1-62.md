# 问题修复和功能优化（V1-62 真实发布取证再降摩擦：`publish:cws` 诊断证据包（JSON）一键落盘）

## 目标
基于 `docs/roadmap_status.md` 的收入优先 Top3，本轮选择 Top1「发布取证证据包（JSON）一键落盘（离线可推进）」作为 1~3 天最小可交付增量主题。

收入第一：真实收入回报路径当前最关键的是“完成一次真实 CWS 发布 + 商店端取证（Top2）”，但其长期阻塞点是网络可达性（代理/VPN + 商店可达）。本轮不依赖外网，优先把既有 `npm run publish:cws -- --dry-run` 的 Proxy Diagnostic + Preflight 从“控制台输出 + 手工 tee”升级为**可一键落盘归档、可审计可复核的诊断证据包（JSON）**，让网络恢复后可以一次性完成真实发布并按统一口径取证，同时在网络不可达时也能沉淀可复制、可定位、可追责的失败证据（避免“做了但无法证明推进了商业化”）。

**所有细节，都由你来定，判断依据是根据核心目标决定应该如何做，而不要停下来问我**

## 具体任务
1、新增 `publish:cws`「诊断证据包（JSON）一键落盘」能力（把发布取证从“文本日志”升级为“结构化证据资产”）
  - 入口命令：`npm run publish:cws -- --dry-run --evidence-dir <dir>`（或等价开关；必须支持指定落盘目录）
  - 行为约束（不得破坏既有发布门禁与退出码语义）：
    - 未提供 `--evidence-dir`：行为保持不变（仅控制台输出，不额外写文件）
    - 提供 `--evidence-dir`：无论 Preflight PASS/FAIL，都必须在退出前落盘证据包文件（FAIL 也要可审计）
    - dry-run：仍严格不进行 upload/publish 网络调用；Preflight FAIL 仍按既有约定退出码为 0（但证据包必须落盘）
    - 非 dry-run：Preflight FAIL 必须阻断真实发布（退出码保持为 1），且证据包必须落盘
  - 证据包 JSON Schema（字段命名/顺序稳定，便于长期归档与 diff；不得包含任何 token/secret）：
    - 必含字段（最小集合）：
      - `packVersion`: 固定为 `v1-62`
      - `exportedAt`: ISO 时间（本地生成）
      - `dryRun`: boolean
      - `extensionVersion`: 以 `manifest.json`/`dist/manifest.json` 一致性校验通过后的版本号为准
      - `zip`: `{ fileName, filePath, bytes, sha256 }`（用于“产物可审计/可复核”，hash 必须可单测）
      - `proxyDiagnostic`: 复用现有 `buildCwsProxyDiagnostic(...)` 的脱敏输出（不得改动脱敏语义）
      - `preflightReport`: 复用现有 `runCwsPreflight(...)` 的结构化报告（不得改动 `reportVersion=v1-47` 语义）
      - `preflightFixHints`: 复用现有 `buildCwsPreflightFixHints(...)` 的最短可执行修复建议数组
      - `credentials`: 仅输出 4 个必需凭据项是否已设置（boolean），不得输出具体值（尤其不得输出 `CWS_CLIENT_SECRET/CWS_REFRESH_TOKEN`）
    - 可选字段（若真实 publish 执行则必须包含；dry-run 必须明确标记为 skipped）：
      - `publishAttempt`: `{ uploaded, published, channel, errorCode, errorMessage }`（错误信息需脱敏；不得包含鉴权 header/response 原文）
  - 文件命名规范（必须稳定、可单测、便于按时间归档；不得包含用户输入/代理明文/网页信息）：
    - 建议：`copylot-cws-publish-diagnostic-pack-<version>-YYYY-MM-DD-HHMMSS.dry-run.json`
    - 建议：`copylot-cws-publish-diagnostic-pack-<version>-YYYY-MM-DD-HHMMSS.publish.json`

2、补齐自动化测试（确保“落盘证据包”长期稳定，不因脚本演进而破坏审计口径）
  - 纯函数抽离建议（可选但推荐，便于单测）：
    - 新增 `scripts/cws-publish-evidence-pack.ts`：集中实现（1）证据包构建（2）文件名生成（3）zip sha256 计算（4）敏感字段屏蔽
  - 更新 `scripts/unit-tests.ts` 覆盖（至少）：
    - 文件名规则：version/dateTime/suffix 正确；不得包含代理明文/用户名密码/campaign 等用户输入
    - Schema 稳定：`packVersion` 固定；必含字段齐全；字段顺序稳定（JSON stringify 后可快照/可 diff）
    - 安全红线：证据包与控制台输出中不得出现 `CWS_CLIENT_SECRET`/`CWS_REFRESH_TOKEN` 原文；`credentials` 仅允许 boolean
    - Preflight FAIL 时：`preflightFixHints` 非空且可执行（至少包含“启用代理/示例 env”类提示）
    - zip hash：对固定输入文件可得到稳定 sha256（避免“做完不可复核”）
  - 必须保证 `bash scripts/test.sh` 全量通过（唯一门禁）
