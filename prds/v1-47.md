## 具体任务
1、升级 `npm run publish:cws` 的代理能力：支持 `socks5://`（以及必要时的 `socks5h://`）代理形态
  - 目标：当环境仅能提供 `socks5://127.0.0.1:PORT`（常见于本地代理/VPN 客户端）时，无需额外“转 HTTP 代理端口”的人工操作即可直连 Google API / CWS API
  - 兼容策略（不破坏现有行为）：
    - 保持对 `HTTP_PROXY/HTTPS_PROXY` 的既有支持
    - 明确代理优先级与覆盖关系（以诊断输出为准），并确保 `--dry-run` 与真实发布使用同一套代理解析逻辑
  - 安全要求：任何诊断输出不得泄露 `CWS_CLIENT_SECRET/CWS_REFRESH_TOKEN` 等敏感信息；若代理 URL 含用户名密码，必须脱敏后再输出

2、新增网络可达性预检（Preflight）+ 错误分类：把“失败原因”变成可复制、可定位、可审计的证据
  - 在上传/发布前对关键外网依赖做最小预检（使用同一代理配置）：
    - `www.googleapis.com`（OAuth/token 相关）
    - `chromewebstore.googleapis.com`（CWS Upload API 相关）
    - （如现有脚本依赖其他域名，以脚本真实依赖为准补齐）
  - 预检输出要求：
    - 输出明确的 PASS/FAIL 与失败类型（DNS、连接超时、TLS、HTTP 状态码等），并生成可直接贴到 Issue 的“Proxy Diagnostic + Preflight Report Block”
    - 当预检失败时：给出最短可执行的修复建议（例如：应设置 `HTTPS_PROXY`、代理 scheme 是否正确、是否需要开启 “use system proxy” 等），避免只给出 `fetch failed`

3、补齐测试/用例与取证落盘（确保可执行、可回归、可用于商业化取证）
  - 新增测试用例文档（固定路径）：`docs/test-cases/v1-47.md`
    - 覆盖：无代理 / `http://` 代理 / `socks5://` 代理三种场景下的预检与 dry-run 输出；以及诊断输出脱敏校验点
  - 自动化测试加固（优先纯函数/可离线跑的单测）：覆盖代理解析、脱敏、预检结果分类等关键逻辑；并保证 `bash scripts/test.sh` 全量通过
  - 新增证据目录与索引（固定路径）：`docs/evidence/v1-47/` + `docs/evidence/v1-47/index.md`
    - 落盘：一次 `npm run publish:cws -- --dry-run` 的诊断输出（含 Proxy Diagnostic Block + Preflight 报告）与文件命名规范，确保可被 v1-45 的发布取证/复盘模板直接引用
  - 新增交付简报（固定路径建议）：`docs/reports/v1-47-report.md`
