## 具体任务
1、完成一次真实 CWS 发布（以“转化入口可用”为准，不做功能迭代）
  - 发布前确认：
    - 确认 CWS 线上当前版本号，并保证本次上传版本号严格递增（避免同版本被拒绝）；如需版本 bump，优先走既有发布流程（`npm run publish`）或按仓库约定手动更新 `manifest.json`（以最终可发布为准）。
    - 发布门禁必须通过：`npm run publish:cws` 会强制执行 `bash scripts/test.sh` + 产物一致性校验 + 重新打包 zip（详见 `scripts/chrome-webstore.ts`）。
  - 执行发布：
    - 运行：`npm run publish:cws`（非 `--dry-run`），上传并 publish 到 `default` channel。
    - 若网络受限，使用 `HTTPS_PROXY`（脚本已支持）确保可达再执行发布。
  - 证据落盘（可审计）：
    - 新增并填写发布简报：`docs/reports/v1-37-report.md`，必须记录：
      - 发布日期、发布版本号、执行命令
      - 关键日志片段（脱敏，不得包含任何 token/secret）
      - CWS 侧验证结果：后台/商店页显示的版本号与发布时间（截图或可复查的证据说明）

2、商店端物料一致性核对（确保“获客/付费意向入口”在店内可见且口径一致）
  - 核对范围（至少覆盖）：
    - 长描述（EN/ZH）与仓库一致：`docs/ChromeWebStore-Description-EN.md`、`docs/ChromeWebStore-Description-ZH.md`
      - 必须包含 “Copylot Pro（Planned/候补）”小节，明确未上线不可用（Not shipped yet）
      - 引导路径必须一致：商店页 -> 扩展内 Options -> Pro Tab -> 加入候补/复制候补文案
      - 必须包含 `docs/monetization/pro-scope.md` 的稳定对外链接（口径可审计）
    - 截图顺序与标题口径：按 `docs/aso/store-assets.md` 的 01-07 顺序（不夸大、不写未实现能力）
    - 教程链接可达：长描述中引用的教程/说明链接能正常打开（用于真实转化承接）
  - 证据落盘：
    - 将“核对清单 + 结果（PASS/FAIL + 备注）”写入 `docs/reports/v1-37-report.md`（与发布证据放在同一份简报中，便于一次审计）

3、从 CWS 安装版本回归“转化入口 + 可导出证据”闭环（证明真实商业化路径可跑通）
  - 安装方式：从 Chrome Web Store 安装本次刚发布的版本（禁止使用 unpacked / 本地加载来替代验收）。
  - 必测路径（收入优先）：
    - 转化入口可见且可用：
      - Popup：存在升级 Pro / 候补入口（与现状一致即可）
      - Options：Pro Tab 可打开，候补入口可打开，且“复制候补文案”可写入剪贴板
    - 商业化证据可导出（隐私合规）：
      - 开启“匿名使用数据”后，触发并导出事件：`pro_entry_opened` / `pro_waitlist_opened` / `pro_waitlist_copied`（以 `docs/growth/telemetry-events.md` 为准），且 `props.source` 取值正确（`popup|options`）
      - 关闭“匿名使用数据”后，事件日志立即清空，且不会补发
  - 测试用例闭环：
    - 新增并填写测试用例文档：`docs/test-cases/v1-37.md`
    - 用例必须覆盖：真实发布（含门禁）、商店物料核对、从商店安装回归（含事件导出审计）、以及一次 `bash scripts/test.sh` 回归记录（日期 + PASS）
