## 用例 A：Pro Tab 出现 handoff 区块

步骤：

1. 打开 `src/options/options.html#pro`。
2. 检查 `#pro-human-approval-handoff-panel`、复制按钮和下载 JSON 按钮。

期望：

- handoff 区块可见。
- 文案明确这是“批准前交接准备”，不是收费实现计划。
- 区块继续标明 `stay_validation` 边界。

## 用例 B：Markdown 摘要包含 blocker、问题与 guardrail

步骤：

1. 点击复制 handoff 摘要。
2. 检查剪贴板文本。

期望：

- 文本包含 `handoff_status=`、`ready_for_human_approval=`、`messaging_boundary=stay_validation`。
- `Approval questions`、`Guardrails`、`Next steps` 三个区段存在。
- 当前 blocker 会同时包含 tracker、payment audit、campaign review 与 messaging guard 聚合结果。

## 用例 C：JSON 产物保持 no-implementation 边界

步骤：

1. 下载 `v4-16` JSON。
2. 读取关键字段。

期望：

- 文件名为 `copylot-pro-human-approval-handoff-v4-16.json`。
- 文件内包含 `blockers`、`approvalQuestions`、`guardrails`、`nextActions`、`sources`。
- guardrail 明确禁止 payment、checkout、subscription 或 collection 实现。

## 用例 D：handoff 聚合逻辑回归

步骤：

1. 运行 `scripts/unit-tests.ts`。
2. 检查 ready 与 blocked 两类分支。

期望：

- tracker 已通过时，handoff 不再重复输出 blocker，并进入 `ready_for_human_approval_review`。
- tracker 未通过时，handoff 会去重聚合 tracker、audit、campaign review 和 surface 级问题。
- 当前样例继续为 `hold_validation`。

## 用例 E：证据与路线图收口

步骤：

1. 复核 `docs/evidence/v4-16/*`、`docs/roadmap.md`、`docs/roadmap_status.md`、`docs/reports/v4-16-report.md`。

期望：

- 证据索引明确这是 handoff 准备包，不是收费实现规划。
- roadmap 把 handoff 包记为新的 S4 进展，但继续保留“tracker 真正放行后才进入收费实现规划”的未完成项。
- 报告说明当前 blocker 仍是 `acquisition_bias_unresolved`、`sample_still_thin` 与同窗未放行。
