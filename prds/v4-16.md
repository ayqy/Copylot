# v4-16 人类批准 handoff 包

## 背景

当 `v4-15` tracker 仍停留在 `hold_validation` 时，仓库里还缺一份能够单独导出的 handoff 包，用来回答“如果以后真的进入人类批准，该看哪些 blocker、哪些问题、哪些 guardrail，以及当前为什么仍不能开始收费实现”。没有这层交接物，就只能靠口头解释当前边界。

## 目标

1. 新增人类批准 handoff 包，汇总 tracker、收费评估审计、跨 `campaign` 复核和话术守门的 blocker、问题、guardrail 与下一步。
2. 明确当前 handoff 只是“批准前交接准备”，不是“收费实现规划”。
3. 为后续真正的人类批准评审固化最小问题集，同时保持 `stay_validation` 外部口径不变。

## 交付范围

1. 新增共享逻辑 `src/shared/pro-human-approval-handoff.ts`。
2. 新增脚本 `scripts/build-pro-human-approval-handoff-pack.ts`，生成 `docs/evidence/v4-16/human-approval-handoff-pack/*`。
3. 更新 `src/options/options.ts`、`src/options/options.html`、`_locales/en/messages.json`、`_locales/zh/messages.json`，支持复制 Markdown 与下载 JSON handoff。
4. 更新 `scripts/unit-tests.ts`、`scripts/ui-integration-tests.ts`、`e2e/options-pro-flow.spec.ts`、`scripts/test.ts`，覆盖 blocker 聚合、guardrail 输出和导出确定性。
5. 补齐 `docs/test-cases/v4-16.md`、`docs/evidence/v4-16/*`、`docs/reports/v4-16-report.md`、`docs/roadmap.md`、`docs/roadmap_status.md`。

## 边界

1. handoff 包只允许输出 blocker、批准问题、guardrail 和下一步，不得承诺收费实现时间线。
2. handoff 包继续限制在本地匿名聚合证据内，不得包含复制内容、URL、标题或联系方式。
3. 即使 handoff 包生成成功，只要 tracker 仍未放行，就不能把它当成收费开发计划或支付实现入口。
