## 自动化入口

```bash
bash scripts/test.sh
```

通过标准：

- 统一测试入口退出码为 `0`
- 代码块悬停复制与整页转换都能稳定输出可直接复用的代码结果
- 缩进、内部空行、围栏代码块输出与现有承诺保持一致
- 可识别行号与复制按钮噪声不会残留在复制结果中
- 商业化证据文档可复核“减少返工 -> 更容易再次复用”的判断链路

## 用例 A：悬停复制代码块时保留缩进与内部空行

步骤：

1. 打开 `e2e/fixtures/code.html` 或等价代码块样例页。
2. 开启 Magic Copy 与 Hover 触发，输出格式设置为 `Plain Text`。
3. 悬停代码块并点击 Copylot 按钮，将结果粘贴到纯文本编辑器或 VS Code。

期望：

- 复制结果保留每一层缩进与内部空行。
- 仅裁剪首尾空行，不压平或重排代码结构。
- 粘贴后可直接继续编辑，不需要人工补缩进。

## 用例 B：可识别行号不会残留到复制结果

步骤：

1. 使用带 `ol.hljs-ln` 或等价可识别结构的代码块样例。
2. 执行悬停复制或块级复制。
3. 检查粘贴结果是否仍含行号文本。

期望：

- 行号不会出现在复制结果中。
- 行号剥离不会误删真实代码内容。
- 同一页面重复复制时结果稳定一致。

## 用例 C：复制按钮噪声整行清理且不误伤真实代码

步骤：

1. 使用顶部或底部带 `Copy`、`复制代码` 等按钮文案的代码块样例。
2. 执行悬停复制并检查输出结果。
3. 再使用正文中真实包含 `Copy` 字样的代码样例进行对照。

期望：

- 仅首尾噪声整行被清理，不做行内子串替换。
- 正文真实代码中的 `Copy`、`clone`、字符串字面量等内容不得被误删。
- 中英文复制按钮噪声都能按白名单规则稳定处理。

## 用例 D：整页转换保留 fenced code 与周边上下文

步骤：

1. 将输出格式设置为 `Markdown`。
2. 在代码密集页面执行整页转换。
3. 检查结果中的说明文字、段落与 fenced code 输出。

期望：

- 周边上下文保留，代码块以 Markdown fenced code 输出。
- 代码块内部仍保留缩进与空行。
- 代码块噪声清理规则与悬停复制路径一致。

## 用例 E：商业化证据可验证“减少返工促进复用”

步骤：

1. 完成至少 1 次代码块悬停复制与 1 次整页转换回归。
2. 生成并检查 `docs/evidence/v4-2/code-block-reuse-audit.md`、`docs/evidence/v4-2/code-block-commercialization-proof.md` 与 `docs/evidence/v4-2/code-block-sample.txt`。
3. 对照 `docs/roadmap_status.md` 的 Top1 目标复核证据口径。

期望：

- 证据文档能说明本轮修复直接降低“复制后还要返工”的成本。
- 证据至少覆盖样例页面、输出对比、回归结果与复用判断口径。
- 证据足以支撑“代码块场景更适合作为第二次 `copy_success` 的复用入口”这一商业化判断。
