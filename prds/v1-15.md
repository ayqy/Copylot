## 具体任务
1、Options 设置页新增“新手引导”入口（稳定 DOM、低打扰、可 i18n）
  - 位置：`src/options/options.html` 的 `#prompts-tab` 顶部（靠近标题/工具栏，默认可见，便于用户发现）
  - 形态：轻量卡片/提示区块（不自动弹窗、不阻断其它操作），包含说明 + 一个按钮
  - 稳定 DOM（用于用例与未来自动化）：
    - 容器：`#options-onboarding-panel`
    - 打开按钮：`#options-onboarding-open`
  - i18n：
    - 新增/补齐入口标题、说明、按钮文案等 key
    - 同步更新 `_locales/en/messages.json` 与 `_locales/zh/messages.json`
    - 确保 `npm run check-i18n` 通过
  - 约束：
    - 不新增任何权限
    - 不新增任何第三方依赖
    - 不新增任何“自动触发”的提示（仅用户点击触发）

2、实现 Options 点击“一键打开引导”逻辑（不影响现有 Options 行为）
  - 逻辑落地：`src/options/options.ts`
  - 打开方式：
    - 点击 `#options-onboarding-open` 后用 `chrome.tabs.create({ url })` 打开扩展内页面
    - URL 必须指向 `src/popup/popup.html`，并携带强制打开引导的标记（推荐 `#onboarding` hash）
      - 例如：`${chrome.runtime.getURL('src/popup/popup.html')}#onboarding`
  - 约束：
    - 不新增任何 telemetry 事件名（允许复用 Popup 现有 `popup_opened/onboarding_*` 行为；开关关闭时本地也不会记录）
    - 点击失败（极端情况下）不得影响 Options 其它功能

3、Popup 支持“deeplink 强制打开引导”（仅命中标记时触发，避免影响现有首次自动弹出逻辑）
  - 逻辑落地：`src/popup/popup.ts`
  - 行为要求：
    - 当 `window.location.hash === '#onboarding'`（或等价判定）时，在初始化完成并可操作后，强制执行 `openOnboardingModal('manual')`
    - 强制打开必须忽略“是否已完成引导”状态（用于回看）
    - 强制打开不应改变既有自动弹出规则：未携带标记时仍按 `shouldAutoShowOnboarding(currentSettings)` 决定是否自动展示
  - 约束：
    - 不新增权限、不引入新依赖
    - 不改变既有引导内容/步骤/埋点事件名

4、补齐测试用例文档（可执行、可回归、可发布）
  - 新增并维护测试/用例文档：`docs/test-cases/v1-15.md`
  - 文档必须覆盖（至少）：
    - 用例 A（入口可见）：打开 Options（默认 `#prompts-tab`），`#options-onboarding-panel` 可见，`#options-onboarding-open` 可点击
    - 用例 B（一键打开）：点击按钮后打开新 Tab，URL 为 `.../src/popup/popup.html#onboarding`，且引导 Modal 自动展示（进度为 1/3）
    - 用例 C（可回看）：在已完成引导的前提下再次从 Options 点击打开，仍会强制展示引导（证明“回看”不受 completed 状态限制）
    - 用例 D（交互回归）：引导的“上一步/下一步/完成/跳过/ESC/点击遮罩关闭”均可用；不会影响 Popup 其它按钮与设置保存
    - 用例 E（状态可审计）：完成/跳过后，`copilot_settings.popupOnboardingCompletedVersion` 与 `popupOnboardingVersion` 关系正确（>=），且后续打开 Popup 不会自动弹出（除非清空配置）
    - 自动化回归：记录一次 `bash scripts/test.sh` 执行结果（命令 + 结论）
  - 验收前完成回归：确保不影响既有 Prompt 管理、Chat 服务、隐私开关、本地日志/增长统计面板、Pro/WOM 入口等功能，达到可发布状态
