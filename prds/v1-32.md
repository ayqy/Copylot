# 问题修复和功能优化（V1-32 发布流程防呆：publish 脚本接入全量回归 + 失败不污染 Git）

## 目标
在不新增权限、不新增任何联网发送、且不改变现有复制/Prompt/引导/口碑/商业化功能逻辑的前提下，补齐 `prds/v1.md` 中“测试用例与发布准备”的最后一段工程化缺口：让 `npm run publish` 成为**可重复、可审计、失败无副作用**的一键发布入口（先全量回归再落盘 commit/tag/zip），并通过可执行的测试用例文档验证，使之达到验收标准、可发布。

## 具体任务
1、改造发布脚本为“先回归、后落盘”的安全顺序（失败无副作用）
  - 修改：`scripts/publish.ts`
  - Preflight（发布前置校验，失败直接退出）：
    - `git status --porcelain` 必须为空；否则提示“工作区不干净，禁止发布”，并退出非 0（避免脚本半途创建 commit/tag）。
    - 读取 `manifest.json` 必须成功，且 `version` 为合法字符串。
  - 版本号更新策略（保持现有“最后一位 +1”的默认规则，但落盘与 git 操作延后）：
    - 计算 `newVersion` 后仍需用户确认（沿用现有交互）；确认后仅更新 `manifest.json` 文件（不立即 commit/tag）。
  - 全量回归（决定是否允许继续发布的唯一门禁）：
    - 运行 `bash scripts/test.sh`（必须包含 lint/type-check/check-i18n/unit-tests/build:prod/verify-prod-build）。
    - 若回归失败：
      - 必须将 `manifest.json` 恢复到原始版本内容（不依赖 `git checkout`，直接写回旧 JSON 并落盘）。
      - 必须退出非 0，且不得创建任何 git commit/tag，不得生成 zip，不得尝试 push / release。

2、发布产物一致性校验与失败兜底提示（保证可审计、可回滚）
  - 修改：`scripts/publish.ts`
  - 仅当 `bash scripts/test.sh` 回归通过后才允许继续：
    - 创建 commit（消息保持：`chore: bump version to ${newVersion}`）
    - 创建 tag（保持：`v${newVersion}`）
    - 打包 `dist/` 为 `plugin-${newVersion}.zip`
  - 打包前一致性校验（失败必须退出非 0，并打印清晰原因）：
    - 读取 `dist/manifest.json`，确保其 `version === newVersion`（防止“版本号已改但 dist 仍是旧产物”导致发版错包）。
  - 后置步骤失败处理（不做任何自动化破坏性回滚，但必须给出可执行的回滚指令）：
    - 任何发生在“commit/tag 已创建”之后的失败（zip/gh release/git push 等），必须在 stderr 输出：
      - 删除本地 tag：`git tag -d v${newVersion}`
      - 回滚 commit（保守建议，避免 `--hard`）：`git reset --soft HEAD~1`（并提示用户自行决定是否需要 `--hard`）

3、补齐发布文档与测试/用例闭环（让验收与回归可执行）
  - 更新发布文档口径：
    - 修改：`BUILD-PRODUCTION.md`
    - 要求：在“构建验证”章节明确发布前的推荐顺序为 `bash scripts/test.sh`（包含 `bash scripts/verify-prod-build.sh`），并保留原有 `find/grep` 作为手工兜底手段。
  - 新增并维护测试/用例文档：`docs/test-cases/v1-32.md`
    - 文档必须覆盖（至少）：
      - 用例 A（失败不污染 Git）：在一个临时分支制造一次可控的 `bash scripts/test.sh` 失败（例如引入可逆的 type-check 失败），运行 `npm run publish` 后应满足：
        - 未产生 commit/tag/zip
        - `manifest.json` 版本号已自动恢复（`git diff` 为空）
      - 用例 B（成功路径产物校验）：在干净工作区且 `bash scripts/test.sh` 可通过时运行 `npm run publish` 并完成到 zip 生成步骤，校验：
        - commit/tag 按约定创建
        - `plugin-${newVersion}.zip` 存在，且 `dist/manifest.json` 版本与 `newVersion` 一致
        - `bash scripts/verify-prod-build.sh` 针对该 `dist/` 自检通过
      - 用例 C（文档一致性）：`BUILD-PRODUCTION.md` 的“构建验证”章节包含 `bash scripts/test.sh` 的推荐入口说明

## 验收标准
