# Copylot Popup 与 Options 隐私 / Pro 页面用户化重设计

## 背景

`Copylot` 当前 popup 首屏强调“30 秒完成第一次干净复制”和多步 onboarding，但交互节奏与信息体量偏强，管理员明确反馈新加的“第一次干净复制”等引导体验很差。与此同时，`src/options/options.html` 的 `privacy` 和 `pro` tab 充满了 telemetry、evidence pack、validation lab、waitlist / distribution / campaign 等开发过程与运营过程细节，并不适合普通插件用户。

## 目标

保留 Copylot 已有能力和脚本绑定的前提下，重做 popup、privacy tab、pro tab 的信息层级与文案：让 popup 更像立即可用的工具面板，让 privacy 说明更像用户信任中心，让 pro tab 更像用户能理解的路线与适用场景说明，而不是内部实验面板。

## ULTRA IMPORTANT 第一原则：任务开始后，不要停下来，直到完成目标
**所有细节，都由你来定，判断依据是根据核心目标决定应该如何做，而不要停下来问我**

## 结合项目代码的需求澄清

Popup 结构位于 `src/popup/popup.html`、`src/popup/popup.css`、`src/popup/popup.ts`，多项交互依赖固定 DOM id，例如 `first-copy-*`、`reuse-primary-*`、`popup-onboarding-*`、`upgrade-pro-entry`。Options 主结构位于 `src/options/options.html`、`src/options/options.css`、`src/options/options.ts`；`privacy` 与 `pro` 标签页中的大量按钮、textarea 与 copy/export 逻辑已在 TS 中绑定具体 id。多语言文案主要来自 `_locales/en/messages.json` 与 `_locales/zh/messages.json`，如果本次改到受 i18n 控制的按钮或标题，需要同步更新对应 key。本次属于维护型 UI 任务，优先保留功能钩子与现有事件逻辑，通过重排、折叠、改写、降级视觉权重的方式降低开发过程感。

## 具体任务

1. 收敛 popup 首屏：弱化或删除过度喧宾夺主的第一次干净复制教学块与 onboarding 入口，让复制动作、快捷 Prompt 复用与基础设置成为更直接的第一层。
2. 重做 `privacy` tab，把“本地处理、不会保存网页内容、可随时清空本地数据”作为主说明；把详细日志、统计与诊断能力降到次级、可折叠、可跳过层级。
3. 重做 `pro` tab，把当前验证素材实验、S3-x 标号、路线验证等内部流程改写为用户能理解的 Pro 方向、适用场景、当前可做与后续路线说明；必要时把内部工具入口折叠为高级资料区。
4. 在不破坏 TS 事件绑定的前提下，尽量保留原有 DOM id 或同步调整脚本，避免删 DOM 后引发空引用。

## 验收标准

* 实现以上所有需求，并测试通过，达到可交互、可运行或可使用的状态。
* 如果本轮任务涉及部署、发布、上线、同步到目标环境或其他交付落地动作，则在测试通过后完成对应动作，并对目标环境效果进行回归验证。

1. Popup 打开后首屏主动作应是立即复制或复用快捷 Prompt，而不是先被大块教学和引导占满。
2. Options 的 `privacy` 和 `pro` tab 不再把 evidence pack、campaign review、validation lab 等内部术语作为主视觉内容。
3. 普通用户能在 10 秒内看懂：Copylot 怎么开始用、数据怎么留在本地、Pro 方向适合什么人。

1、以上任务规模非常大，每一个都很有挑战性，弄清楚需求后你应该创建新的需求文档来记录每个子需求，例如x-1.md、x-2.md等等（x表示当前的需求代号v几中的几）
2、一步一步来，对于每个阶段，边执行边更新对应文档中的进度
3、根据需求创建测试用例文档，通过单元测试和自动化测试来确保代码的正确性
4、目标任务成功完成之后，写一份简要的汇报材料文档（用于口头汇报，所以要简明扼要抓住重点），说清楚每一个具体任务现在是什么状态（是否按要求完成了），已经实现了什么效果，以及本次对代码的修改范围（涉及哪些目录和文件）
5、如果涉及依赖服务、外部系统、发布流程或其他高风险变更，你一定要谨慎修改；测试通过后再执行发布或变更，并保留必要的回滚意识与验证步骤，避免引发线上故障或连带问题。
