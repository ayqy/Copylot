# V1-6 商业化意向验证 MVP（升级 Pro / 候补名单入口）

## 目标
实现“升级 Pro / 加入候补名单”最小闭环：在不新增权限、不影响现有免费功能、且不破坏“本地处理/隐私优先/零打扰”原则的前提下，Popup 与 Options 提供可发现入口与价值说明，并可一键跳转到候补登记页面；同时在“匿名使用数据”开关开启时记录本地匿名意向事件，用于评估转化意向，使之达到验收标准并可发布。

## ULTRA IMPORTANT 第一原则：任务开始后，不要停下来，直到完成目标
**所有细节，都由你来定，判断依据是根据核心目标决定应该如何做，而不要停下来问我**

## 具体任务
1、升级 Pro / 候补名单入口（零打扰、可回归、可 i18n、稳定 DOM）
  - Popup 增加“升级 Pro”入口（不自动弹窗、不做强引导）：
    - 入口 DOM 必须稳定：`#upgrade-pro-entry`
    - 点击行为：打开 Options 页并定位到 Pro Tab（建议使用 `src/options/options.html#pro`）；不阻断用户现有操作
    - Popup 点击后必须记录本地匿名事件：`pro_entry_opened`（props：`{ source: 'popup' }`，仅在开关开启时记录）
  - Options 增加 Pro Tab 与价值说明（用于形成最小“入口 -> 价值说明 -> 行动”链路）：
    - Tab 按钮：`.tab-btn[data-tab="pro"]`；内容容器：`#pro-tab`
    - 支持 URL hash 定位：当打开 `src/options/options.html#pro` 时，默认激活 Pro Tab（需在 `src/options/options.ts` 落地）；当用户手动切换到 Pro Tab 时也应正常显示
    - 内容至少包含：
      - Pro 价值说明（短文 + 3~5 条能力方向要点，来源可参考 `docs/roadmap.md` 的“商业化 MVP（Pro 试水）”）
      - “加入候补名单”按钮：`#pro-waitlist-button`
      - “复制候补文案”按钮：`#pro-waitlist-copy`
    - 用户打开 Pro Tab 时必须记录本地匿名事件：`pro_entry_opened`（props：`{ source: 'options' }`，仅在开关开启时记录）
  - i18n：补齐 `_locales/en/messages.json` 与 `_locales/zh/messages.json` 对应文案 key，并确保 `npm run check-i18n` 通过
  - 建议实现位置（用于落地与 code review 对齐）：
    - Popup：`src/popup/popup.html`、`src/popup/popup.ts`、`src/popup/popup.css`
    - Options：`src/options/options.html`、`src/options/options.ts`、`src/options/options.css`
  - 约束：
    - 不新增任何权限
    - 不实现任何 Pro 功能/支付/订阅，仅提供入口与候补登记
    - 不收集用户复制内容/页面内容

2、候补登记链接（可审计、可维护、默认不含任何用户内容）
  - 新增共享模块：`src/shared/monetization.ts`（或同等语义文件），用于集中管理“候补登记”链接构造与模板：
    - 候补登记目标：GitHub `issues/new`（基准 URL 可固定为仓库 Issues 新建页，必须可审计）
    - 提供 `buildProWaitlistIssueUrl(...)`：通过 URL query 预填 `title/body`（body 中仅允许包含环境信息与占位提示，不得包含任何用户复制内容/页面内容/URL/标题）
      - 环境信息至少包含：扩展版本（`chrome.runtime.getManifest().version`）、扩展 ID（`chrome.runtime.id`）、系统语言（`navigator.language` + `chrome.i18n.getUILanguage()`）
    - body 必须包含占位段落：使用场景/期望的 Pro 能力/联系方式（可选），并明确提示“不要填写任何网页内容或复制内容”
  - Popup 与 Options 必须复用该函数，避免 hardcode 分散在多个入口

3、本地匿名意向埋点 + 文档/用例（可执行、可回归）
  - Telemetry 事件（仅在“匿名使用数据”开关开启时写入 `copilot_telemetry_events`）：
    - `pro_entry_opened`，props：`{ source: 'popup' | 'options' }`
    - `pro_waitlist_opened`，props：`{ source: 'popup' | 'options' }`
    - `pro_waitlist_copied`，props：`{ source: 'options' }`
  - 更新 `src/shared/telemetry.ts`：
    - 追加事件名 union 与 props allowlist
    - 保证 props 仍严格白名单过滤，且值类型仅 `string|number|boolean`
  - 更新埋点清单：`docs/growth/telemetry-events.md` 增补上述事件（触发位置/props/隐私约束/保留策略）
  - 单测最小补齐：在 `scripts/unit-tests.ts` 增加用例，覆盖新增事件的 `sanitizeTelemetryEvent` props 白名单过滤行为
  - 新增并维护测试/用例文档：`docs/test-cases/v1-6.md`
    - 必须覆盖（至少）：
      - Popup/Options 入口可见；Popup 点击可打开 Options 并定位到 Pro Tab
      - “加入候补名单”跳转可用；GitHub Issue 页面已预填 title/body；body 仅包含环境信息与占位提示，不包含复制内容/页面内容/URL/标题
      - 匿名使用数据默认关闭：点击入口不产生 `copilot_telemetry_events`
      - 开关开启：点击入口产生 `pro_*` 事件；props 字段符合白名单；日志最多 100 条且先进先出
      - 关闭开关：立即清空 `copilot_telemetry_events`；关闭后再次点击不记录（不发送、不缓存、不补发）
      - 回归：不影响现有复制/Prompt/引导/评价引导/分享/反馈等功能
    - 验收前必须按文档执行手工测试，并记录 `bash scripts/test.sh` 全量通过（lint/type-check/check-i18n/unit-tests/build:prod）

## 验收标准
* 修复上述所有问题，并测试通过，达到可发布状态
* 实现以上所有需求，并测试通过，达到可发布状态
