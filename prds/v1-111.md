# 第一次干净复制承接口径对齐与首屏转化基线补齐

## 目标
把当前已上架版本的安装承接面收敛为围绕“第一次干净复制”的单一叙事：统一 Popup 首屏、Chrome Web Store 说明与截图顺序、官网首页承接规范和安装指引，并补齐可执行测试与可审计证据，使本轮交付可以被验证为真实推动安装后首次 `copy_success` 与可转化获客增长。

## ULTRA IMPORTANT 第一原则：任务开始后，不要停下来，直到完成目标
**所有细节，都由你来定，判断依据是根据核心目标决定应该如何做，而不要停下来问我**

## 具体任务
1、统一 Popup 首屏与第一次成功引导
  - 更新 `src/popup/popup.html`、`src/popup/popup.ts`、`src/popup/popup.css` 的首屏价值主张、快捷路径与引导文案，让用户打开扩展后立刻知道“先复制什么、怎么触发、复制后贴到哪里”。
  - 首屏和引导必须以长文、表格、代码块三类场景中的第一次干净复制为核心，不新增与表单、问卷、候补或支付相关的 CTA。
  - 若需要同步行为埋点，仅限复用现有 `src/shared/growth-stats.ts` 与 `src/shared/telemetry.ts` 的首次成功字段，不在本轮扩张新的 Pro 或留资事件模型。

2、统一商店说明、截图顺序、官网首页承接规范与官方链接资产
  - 更新 `docs/ChromeWebStore-Description-ZH.md`、`docs/ChromeWebStore-Description-EN.md`、`docs/ChromeWebStoreSetup.md`、`src/shared/external-links.ts`，把商店文案、安装指引、CTA 与对外链接统一成“30 秒完成第一次干净复制”叙事。
  - 新增 `docs/growth/install-first-launch-pack-v1-111.md` 与 `docs/evidence/v1-111/` 证据目录，至少包含 `index.md`、`official-links.json`、`screenshot-sequence.md`、`homepage-alignment-spec.md`、`first-copy-install-guide.md`。
  - 官网源码仍不在当前仓库时，本轮不新建站点工程，只输出可直接交给站点仓执行的首页承接规范、截图顺序与安装指引资产。

3、补齐首次成功承接的测试、审计与汇报闭环
  - 新增 `docs/test-cases/v1-111.md`，覆盖 Popup 首屏、商店说明、外链、安装指引、截图顺序与无表单边界，并以 `bash scripts/test.sh` 为统一回归入口；按需更新 `e2e/popup-growth-flow.spec.ts`、`e2e/options-pro-flow.spec.ts` 与相关单测。
  - 在 `docs/evidence/v1-111/` 下补齐 `funnel-audit-spec.md`，明确安装 -> 首次 Popup 打开 -> 首次 `copy_success` 的导出路径、对账方式与人工复核步骤，保证本轮承接优化可被审计。
  - 交付完成后输出 `docs/reports/v1-111-report.md`，说明三项任务状态、商业化推进效果与实际修改范围。

## 验收标准
* 修复上述所有问题，并测试通过，达到可发布状态
* 实现以上所有需求，并测试通过，达到可发布状态
* `docs/test-cases/v1-111.md` 明确给出手工步骤、自动化入口与通过标准，并可直接对应 `bash scripts/test.sh` 以及本轮涉及的 Popup / E2E 回归。
* `src/popup/popup.html`、`src/popup/popup.ts`、`docs/ChromeWebStore-Description-ZH.md`、`docs/ChromeWebStore-Description-EN.md`、`docs/evidence/v1-111/screenshot-sequence.md` 的主标题、前三个卖点、截图顺序和安装指引一致，且第一目标都是“第一次干净复制”。
* `docs/evidence/v1-111/official-links.json`、`docs/growth/install-first-launch-pack-v1-111.md` 与扩展内对外链接只保留官网首页、商店安装页、隐私说明页、Pro 路线页四类固定入口，不出现表单、问卷、候补或支付承接。
* `docs/evidence/v1-111/homepage-alignment-spec.md` 与 `docs/evidence/v1-111/first-copy-install-guide.md` 足够作为官网首页承接的可执行交付物，即使站点源码不在当前仓库，也能直接支持站点仓同步实现。
* `docs/evidence/v1-111/index.md` 与 `docs/evidence/v1-111/funnel-audit-spec.md` 提供可验证的商业化证据：评审人可以据此复核安装入口、截图承接、首次成功引导和安装 -> `copy_success` 导出路径，能够证明本轮推进了真实安装转化或可转化获客增长，而不是仅完成文案整理。

## 注意
1、以上任务规模非常大，每一个都很有挑战性，弄清楚需求后你应该创建新的需求文档来记录每个子需求，例如x-1.md、x-2.md等等（x表示当前的需求代号v几中的几）
2、一步一步来，对于每个阶段，边执行边更新对应文档中的进度
3、根据需求创建测试用例文档，通过单元测试和自动化测试来确保代码的正确性

5、本轮默认测试用例路径为 `docs/test-cases/v1-111.md`，证据路径为 `docs/evidence/v1-111/*`，汇报路径为 `docs/reports/v1-111-report.md`；如果执行过程中发现需要改动真实官网页面源码，仍以当前仓库内的承接规范与证据包为主交付，不偏离 1-3 天的最小增量范围。
