# V1-95 CWS 代理就绪门禁与发布取证最小闭环 测试用例

## 目标
验证 `prds/v1-95.md` 要求：`publish:cws` 在代理未启用时能稳定输出 `proxy_not_started` 与 `pxy` 修复动作；诊断证据包新增 `proxyReadiness.status/fixCommand/blocking` 且保持 `packVersion=v1-62` 兼容。

## 前置说明
- 本环境 `CWS_*` 凭据已配置。
- `pxy` 命令定义在 bash profile；若当前 shell 未加载，需先执行 `source ~/.bash_profile`。

## 用例 A：未启用代理时 dry-run 命中 `proxy_not_started` 并给出修复动作
步骤：
1. 清理代理变量：`unset CWS_PROXY HTTPS_PROXY HTTP_PROXY ALL_PROXY https_proxy http_proxy all_proxy`
2. 执行：`npm run publish:cws -- --dry-run --evidence-dir docs/evidence/v1-95/preflight/`
3. 断言控制台输出包含：
   - `proxyReadiness.status）：proxy_not_started`
   - `预检阻塞分类：proxy_not_started`
   - `修复动作 1/2：pxy`
   - `修复动作 1/2（bash profile）：source ~/.bash_profile && pxy`
   - `修复动作 2/2：npm run publish:cws -- --dry-run`
4. 断言证据包新增字段：
   - `proxyReadiness.status === "proxy_not_started"`
   - `proxyReadiness.blocking === true`
   - `proxyReadiness.fixCommand` 包含 `source ~/.bash_profile && pxy`

期望：
- 即使网络不可达，仍能给出可复制修复路径，不再只报失败。

## 用例 B：启用代理后 dry-run 命中 `proxyReadiness.status=ready`
步骤：
1. 启动代理：`pxy`（或 `source ~/.bash_profile && pxy`）
2. 重新执行：`npm run publish:cws -- --dry-run --evidence-dir docs/evidence/v1-95/preflight/`
3. 断言：
   - 控制台输出 `proxyReadiness.status）：ready`
   - 证据包 `proxyReadiness.status === "ready"`
   - 若仍失败，失败分类应为非 `proxy_not_started`（用于区分权限/账号等非代理阻塞）

期望：
- 可以区分“代理已就绪”和“非代理阻塞”。

## 用例 C：证据包 JSON 可解析且新增字段口径稳定
步骤：
1. 选择最新证据包：`docs/evidence/v1-95/preflight/*.json`

   - `node -e "const fs=require('fs');const p=process.argv[1];const j=JSON.parse(fs.readFileSync(p,'utf8'));console.log(j.packVersion,j.proxyReadiness.status,j.proxyReadiness.blocking,Boolean(j.proxyReadiness.fixCommand));" <evidence-file>`

   - `proxyReadiness` 存在且包含 `status/fixCommand/blocking`
