# 问题修复和功能优化（V1-95 CWS 代理就绪门禁与发布取证最小闭环）

## 目标
在不改变免费核心功能、不引入新权限的前提下，优先打通“商店端可执行取证”这一真实收入回报路径关键前置：把 `publish:cws` 从“网络失败后人工猜测原因”升级为“可自动识别代理未启用、可一键指路 `pxy`、可导出证据包复核”的最小闭环，使 S0 的 Listing 同步与商店端转化取证可以稳定推进。

## ULTRA IMPORTANT 第一原则：任务开始后，不要停下来，直到完成目标
**所有细节，都由你来定，判断依据是根据核心目标决定应该如何做，而不要停下来问我**

## 具体任务
1、发布前代理就绪门禁：识别“代理未启用”并给出可执行修复指令
  - 修改：`scripts/cws-proxy.ts`、`scripts/cws-preflight.ts`、`scripts/chrome-webstore.ts`
  - 要求：
    - 在 `publish:cws` 预检阶段新增稳定分类：当 CWS 目标网络不可达且未命中可用代理配置时，输出 `proxy_not_started`（或等价稳定枚举）
    - 诊断输出必须包含可直接复制的修复动作：先执行 `pxy`（必要时附 `source ~/.bash_profile && pxy`）再重试 `npm run publish:cws -- --dry-run`
    - 不输出敏感信息：仅允许输出脱敏后的代理信息与布尔状态，不得回显 token/secret 原文

2、诊断证据包口径升级：把“代理就绪状态”固化到可审计 JSON
  - 修改：`scripts/cws-publish-evidence-pack.ts`、`scripts/chrome-webstore.ts`
  - 要求：
    - 在诊断证据包新增稳定字段（示例口径）：`proxyReadiness.status`、`proxyReadiness.fixCommand`、`proxyReadiness.blocking`
    - 支持通过 `npm run publish:cws -- --dry-run --evidence-dir docs/evidence/v1-95/preflight/` 直接落盘并复核上述字段
    - 字段必须兼容既有 `packVersion=v1-62` 证据消费逻辑（不破坏现有解析链路）

3、测试与用例门禁落地（确保 1-3 天内可发布）
  - 修改：`scripts/unit-tests.ts`、`docs/test-cases/v1-95.md`
  - 自动化要求：
    - 单测覆盖“无代理 -> `proxy_not_started` 分类 + `pxy` 修复指令存在”
    - 单测覆盖“已配置代理 -> `proxyReadiness.status=ready`（或等价状态）”
    - `bash scripts/test.sh` 全量通过
  - 用例文档要求（固定路径）：`docs/test-cases/v1-95.md`
    - 用例 A：未启用代理时执行 dry-run，验证诊断块与证据包出现 `proxy_not_started` 与修复指令
    - 用例 B：启用代理后执行 dry-run，验证代理就绪状态字段正确
    - 用例 C：验证证据包可 `JSON.parse` 且新增字段口径稳定（含一次回归记录：PASS/BLOCKED + 证据文件名）

## 验收标准
* 修复上述所有问题，并测试通过，达到可发布状态
* 实现以上所有需求，并测试通过，达到可发布状态
* 可验证的商业化推进证据（必须可审计/可复盘）：
  - 在“代理未启用”场景执行 `npm run publish:cws -- --dry-run --evidence-dir docs/evidence/v1-95/preflight/` 时，输出明确阻塞分类与 `pxy` 修复动作，不再出现“仅失败无指路”
  - 证据包中可稳定导出 `proxyReadiness.*` 字段，能支撑 S0 后续 `v1-70/v1-71` 的商店端取证执行判断
  - 当代理已启用并重跑 dry-run 后，预检结果可区分“代理链路已就绪”与“权限/账号等非代理阻塞”，支持收入链路问题快速归因
* 测试与用例可执行：
  - `docs/test-cases/v1-95.md` 可逐条执行，并包含一次回归记录（PASS/BLOCKED + 证据文件名）
  - `bash scripts/test.sh` 全量通过（含新增单测）

## 注意
1、以上任务是“最小可交付增量（1-3 天）”，只解决“代理就绪判定 + 证据固化 + 指路降摩擦”，不扩展到支付、订阅或新增长功能
2、收入第一：本轮目标是提升“商店端取证与 Listing 同步”执行成功率，确保转化入口优化具备可落地、可验证、可复盘的前置条件
3、安全红线：不得泄露 `CWS_CLIENT_SECRET`、`CWS_REFRESH_TOKEN` 等敏感值；日志与证据包仅允许脱敏信息
4、若 `pxy` 可用但 CWS 权限仍不足，需在 `docs/roadmap_status.md` 的“阻塞与需要的人类输入”明确标注“账号/权限阻塞”，并顺延执行下一优先级工作项
5、目标任务完成后，补齐简要汇报材料（固定路径：`docs/reports/v1-95-report.md`），说明阻塞分类命中率、证据包字段变化、以及对 S0 收入链路推进的实际效果
